아래 절차에 따라 현재 변경사항을 커밋해 주세요.

> 이 저장소는 husky git hook이 설정되어 있어 커밋 시 아래 검증이 자동으로 실행됩니다.
>
> - `pre-commit`: 스테이징된 파일을 Biome으로 검사합니다. 위반이 있으면 커밋이 차단되며 **자동 수정하지 않습니다**.
> - `commit-msg`: 커밋 메시지가 아래 형식과 타입을 따르는지 검증합니다. 위반 시 커밋이 차단됩니다.
>
> 훅을 우회하는 `--no-verify`(`-n`)는 사용하지 않습니다. 검증에 걸리면 원인을 고친 뒤 다시 커밋하세요.

## 1단계: 변경사항 파악

`git status`와 `git diff`를 실행하여 staged/unstaged 변경사항을 모두 파악하세요.

## 2단계: 논리적 단위로 분류

변경사항을 아래 타입 기준에 맞춰 논리적 단위로 그룹화하세요.
서로 관련 없는 변경사항은 반드시 별도 커밋으로 분리합니다.

| 타입       | 사용 시점                          |
| ---------- | ---------------------------------- |
| `feat`     | 새로운 기능 추가                   |
| `fix`      | 버그 수정                          |
| `docs`     | 문서 변경 (README, 주석 등)        |
| `style`    | 코드 동작에 영향 없는 포맷 변경    |
| `refactor` | 동작 변경 없는 코드 리팩토링       |
| `chore`    | 빌드 스크립트, 패키지 설정 등 잡무 |
| `perf`     | 성능 개선                          |
| `ci`       | CI 설정 또는 스크립트 수정         |
| `release`  | 버전 릴리즈 또는 태깅              |

## 3단계: 커밋 실행

각 논리적 단위마다 아래 형식으로 커밋하세요.

**커밋 메시지 형식:**

```
<타입>(선택: 범위): <제목>

(선택) 본문 — 변경 이유와 내용을 간략히

(선택) 푸터 — 이슈 번호, breaking change 등
```

**규칙:**

- 타입은 위 표의 값(`feat`, `fix`, `docs`, `style`, `refactor`, `chore`, `perf`, `ci`, `release`) 중 하나를 사용합니다. `commit-msg` 훅이 이 형식을 검증합니다.
- 제목은 한국어로 작성합니다.
- 제목 끝에 마침표를 붙이지 않습니다. (마침표가 있으면 `commit-msg` 훅이 차단합니다)
- 민감한 파일(.env, credentials 등)은 절대 커밋하지 않습니다.
- `git add [관련 파일]`로 파일을 선택적으로 스테이징합니다.

**훅이 커밋을 차단하면:**

- `pre-commit`(Biome)에서 막히면 `bun run lint`로 위반을 확인하고, 필요한 파일만 직접 수정한 뒤 다시 스테이징해 커밋합니다. (`bun run format`은 자동 수정이므로 의도한 파일에만 신중히 사용)
- `commit-msg`에서 막히면 출력된 형식 안내에 맞게 메시지를 고쳐 다시 커밋합니다.

**예시:**

```
feat(auth): 카카오 소셜 로그인 기능 추가
fix(trip): 일정 저장 시 날짜 오류 수정
docs: CONTRIBUTING.md 기여 절차 추가
chore: biome 린터 설정 추가
```

## 4단계: 완료 확인

`git log --oneline -10`으로 커밋 이력을 확인하고,
`git status`로 남은 변경사항이 없는지 검증하세요.

> 참고: 이후 `git push` 시 `pre-push` 훅이 `bun run test`를 실행하며, 테스트가 실패하면 push가 차단됩니다.
