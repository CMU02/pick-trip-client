# 예외 처리 흐름

## 서버 오류 응답 계약

서버 API 예외는 다음 형태를 기본으로 가정한다.

```json
{
  "code": "DOMAIN_ERROR_TYPE",
  "message": "사용자에게 표시할 한국어 메시지",
  "traceId": "request-trace-id"
}
```

## 클라이언트 처리 원칙

- 사용자에게는 `message`를 우선 표시한다.
- `traceId`가 있으면 오류 상세 또는 문의 안내에 포함할 수 있다.
- 서버 내부 예외 메시지, stack trace, raw response body를 그대로 노출하지 않는다.
- 네트워크 오류와 서버 도메인 오류를 구분한다.
- 사용자가 복구할 수 있는 오류에는 다음 행동을 함께 제공한다.

## 오류 유형별 UI

| 상황 | 처리 |
| --- | --- |
| 네트워크 실패 | 재시도 버튼과 연결 상태 안내 |
| 400 검증 오류 | 잘못된 입력 필드 근처에 메시지 표시 |
| 401 인증 필요 | 로그인 유도 |
| 403 권한 없음 | 접근 불가 안내 |
| 404 데이터 없음 | 빈 상태 또는 링크 만료 안내 |
| 409 상태 충돌 | 최신 상태 새로고침 안내 |
| 500 서버 오류 | 잠시 후 재시도 안내 |

## 일정 생성 오류

일정 생성은 사용자가 많은 입력을 완료한 뒤 호출하는 핵심 액션이므로 실패 시 입력값을 유지한다.

- 선택한 콘텐츠 목록을 초기화하지 않는다.
- 여행 조건 폼 값을 유지한다.
- 실패 원인이 콘텐츠 부족, 날짜 오류, 서버 과부하 중 무엇인지 가능한 범위에서 구분한다.
- 재시도 버튼을 제공한다.

## 로깅 기준

- 개발 환경에서는 console 또는 Next.js fetch logging을 활용할 수 있다.
- 운영 환경에서는 사용자 개인정보, 토큰, 위치 민감 정보를 로그에 남기지 않는다.
- traceId가 있다면 오류 추적 키로 사용한다.
