# 정보 보안 보완 방안

## 클라이언트 보안 원칙

- 브라우저에 전달되는 모든 값은 공개될 수 있다고 가정한다.
- `NEXT_PUBLIC_` 환경변수에는 공개 가능한 값만 넣는다.
- OAuth client secret, JWT signing key, DB 비밀번호, 서버 API key는 클라이언트 저장소와 브라우저 번들에 포함하지 않는다.
- 인증 토큰 저장 방식은 서버 정책과 브라우저 보안 요구사항을 함께 검토한다.

## 환경변수

| 유형 | 사용 가능 여부 |
| --- | --- |
| `NEXT_PUBLIC_API_BASE_URL` | 가능 |
| OAuth client ID | 공개용이면 가능 |
| OAuth client secret | 금지 |
| JWT secret | 금지 |
| DB URL / 계정 / 비밀번호 | 금지 |

## API 호출

- 인증이 필요한 요청은 토큰 만료와 재로그인 흐름을 처리한다.
- 401 응답을 무한 재시도하지 않는다.
- 사용자의 입력값은 서버 검증을 전제로 하되, 클라이언트에서도 기본 검증을 수행한다.
- 서버 오류 응답을 화면에 그대로 출력하지 않는다.

## XSS 방지

- 서버에서 받은 HTML 문자열을 그대로 렌더링하지 않는다.
- `dangerouslySetInnerHTML`은 기본적으로 사용하지 않는다.
- 외부 이미지와 링크는 출처를 검토하고, 새 창 링크에는 `rel="noopener noreferrer"`를 적용한다.
- 사용자 생성 텍스트는 React의 기본 escape 동작을 유지한다.

## 개인정보

- 여행 조건, 선택 콘텐츠, 위치 관련 정보는 필요한 화면과 API에만 전달한다.
- 로컬 스토리지에 개인정보나 장기 인증 토큰을 저장하지 않는다.
- 공유 일정 화면에는 공개 가능한 정보만 노출한다.

## 의존성 관리

- 패키지 추가 시 목적과 대체 가능성을 PR에 적는다.
- UI 편의를 위해 대형 라이브러리를 추가하기 전에 번들 크기와 유지보수성을 검토한다.
- `bun.lock` 변경은 의존성 변경과 함께 리뷰한다.
